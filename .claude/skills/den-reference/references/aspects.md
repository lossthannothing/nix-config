# Aspects Reference

## `den.aspects`

An attribute set of aspects. Each aspect key names an aspect; its value contains per-class deferred modules and optional `includes` and `provides`.

```nix
den.aspects.dns = {
  nixos.services.resolved.enable = true;
  darwin.networking.dns = [ "1.1.1.1" ];
  includes = [ ./dns ];
};
```

Den auto-generates `den.aspects` entries from `den.hosts`, `den.homes`, and `den.users`. For every declared host/home/user, an aspect is created with the appropriate class configurations.

## Aspect structure

| Key | Purpose |
|-----|---------|
| `<class>` (e.g. `nixos`, `homeManager`) | Config merged into hosts/homes of that class |
| `includes` | List of modules or functions dispatched by context |
| `__functor` | Auto-generated by `parametric`; drives dispatch |

## `den.ful` â€” Namespaced aspects

Type: `attrsOf aspectsType`. Namespaced aspect collections. Each key is a namespace name, each value is a full `aspectsType`. Populated by `den.namespace` or by merging upstream `denful` flake outputs.

```nix
den.ful.myns = {
  some-aspect = { nixos.services.foo.enable = true; };
};
```

## `den.raw` â€” Raw flake output

Type: `attrsOf raw`. Raw flake output for publishing namespaces. Set automatically by `den.namespace`; consumed by downstream flakes that import your aspects.

## Static vs parametric includes

Functions in `includes` receiving `{ class, aspect-chain }` are **static** â€” evaluated once during aspect resolution. Functions receiving context arguments (`{ host }`, `{ user }`, etc.) are **parametric** â€” evaluated per context during `ctxApply`.

```nix
# Static â€” runs once per class
includes = [ ({ class, ... }: { nixos.services.foo.enable = class == "nixos"; }) ];

# Parametric â€” runs per host
includes = [ ({ host }: { nixos.networking.hostName = host.hostName; }) ];
```

## `meta.adapter` â€” Aspect filtering

An aspect can declare a subtree adapter that controls how its includes are resolved. The adapter composes with the inherited adapter from parent aspects or context nodes.

```nix
den.aspects.monitoring = {
  meta.adapter = inherited: den.lib.aspects.adapters.filter (a: a.name != "noisy-exporter") inherited;
  includes = [
    den.aspects.prometheus
    den.aspects.noisy-exporter  # filtered out
    den.aspects.grafana
  ];
};
```

The adapter receives the inherited adapter and returns a new one. A parent's filter cannot be overridden by children.

Context-level adapters apply transitively to all aspects resolved within that context:

```nix
den.ctx.host.meta.adapter = inherited: den.lib.aspects.adapters.filter (a: a.name != "debug-tools") inherited;
```

## `meta.provider` â€” Origin tracking

Tracks the structural origin of an aspect as a path. Top-level aspects have `meta.provider = []`. An aspect provided by `foo` (via `foo.provides.bar`) has `meta.provider = ["foo"]`. Deeply nested: `foo._.bar._.baz` has `meta.provider = ["foo" "bar"]`.

Adapters can use this to distinguish aspects by origin:

```nix
meta.adapter = inherited: den.lib.aspects.adapters.filter (a: lib.take 1 (a.meta.provider or []) == ["monitoring"]) inherited;
```

## Cross-context forwarding

Entities expose `.resolved` â€” the aspect from their context pipeline. `den._.forward` can pull configuration from one entity into another:

```nix
den.aspects.iceberg.includes = [
  ({ host }: den._.forward {
    each = lib.filter (h: h != host) (lib.attrValues den.hosts.${host.system});
    fromClass = _: "ssh-host-key";
    intoClass = _: host.class;
    intoPath = _: [ ];
  })
];
```

When `den._.forward` is called without `fromAspect`, it defaults to `item.resolved`.

## Resolution pipeline

When aspects are resolved for a host, Den:

1. Collects all aspects referenced by the host
2. Extracts the class-specific config (e.g. `nixos` for NixOS hosts)
3. Evaluates static includes with `{ class, aspect-chain }`
4. Builds context pairs from `den.ctx`
5. Applies parametric includes via `ctxApply`
6. Merges everything into the host's `evalModules` call
